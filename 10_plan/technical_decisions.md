## ADR-001: 工作流引擎选型

### 状态
✅ **已批准**

### 上下文
需要选择工作流编排工具，串联多个 Agent 完成内容生产流程。

| 考虑因素 | 要求 |
|---------|------|
| 可视化编辑 | 降低维护成本 |
| 自托管 | 数据隐私 |
| Python 支持 | 与 Skills 集成 |
| 成本 | 开源优先 |

### 考虑的选项

#### 选项 A: n8n
| 优点 | 缺点 |
|-----|------|
| ✅ 开源，可自托管 | 学习曲线 |
| ✅ 可视化节点编辑 | - |
| ✅ 支持 Python 代码 | - |
| ✅ 丰富的集成 | - |
| ✅ 活跃的社区 | - |

#### 选项 B: Make.com
| 优点 | 缺点 |
|-----|------|
| ✅ 功能强大 | ❌ 仅云端，需付费 |
| ✅ 界面精美 | ❌ 数据不在本地 |
| ✅ 大量集成 | ❌ 成本高 |

#### 选项 C: CrewAI / LangGraph
| 优点 | 缺点 |
|-----|------|
| ✅ 原生 Python | ❌ 纯代码，无可视化 |
| ✅ Agent 原生 | ❌ 学习曲线陡峭 |
| ✅ 灵活强大 | ❌ 调试困难 |

### 决策
**选择 n8n**

### 理由
1. 用户在需求澄清中已确认使用 n8n
2. 可视化界面便于后期维护和调整
3. 开源自托管，数据可控
4. 与 Claude Code Skills 可通过 HTTP Request Agent 集成

### 后果
- 需要部署 n8n 实例（本地或 Docker）
- Agent 间通信通过 n8n Workflow 编排
- Skills 需要提供 HTTP 接口供 n8n 调用

## ADR-002: Agent 框架选型

### 状态
✅ **已批准**

### 上下文
需要选择 Agent 实现框架，构建各个内容处理 Agent。

| 考虑因素 | 要求 |
|---------|------|
| 与 Claude Code 集成 | 用户已有经验 |
| 可编程性 | 支持复杂逻辑 |
| 模块化 | 每个 Agent 独立 |

### 考虑的选项

#### 选项 A: Claude Code Skills
| 优点 | 缺点 |
|-----|------|
| ✅ 用户已有经验 | - |
| ✅ 与 Claude Code 深度集成 | - |
| ✅ 可编程，支持文件操作 | - |
| ✅ 模块化设计 | - |

#### 选项 B: CrewAI
| 优点 | 缺点 |
|-----|------|
| ✅ Agent 原生支持 | ❌ 与 Claude Code 独立 |
| ✅ Role-playing 定义 | ❌ 需要额外集成工作 |

#### 选项 C: LangGraph
| 优点 | 缺点 |
|-----|------|
| ✅ 状态机支持 | ❌ 复杂度高 |
| ✅ 可视化调试 | ❌ 学习曲线陡 |

### 决策
**选择 Claude Code Skills**

### 理由
1. 用户已有 Claude Code 和 Skills 使用经验
2. 无需额外框架，降低复杂度
3. n8n 通过 HTTP Request 调用 Skills
4. 每个Skill对应一个Agent角色

### 架构示意
```
┌─────────────────────────────────────────────────────────────┐
│  n8n Workflow                                               │
│                                                             │
│  [Webhook] → [HTTP Request: geektime-collector]            │
│       ↓                                                     │
│  [HTTP Request: tech-explorer-writer]                      │
│       ↓                                                     │
│  [HTTP Request: wechat-formatter]                          │
│       ↓                                                     │
│  [HTTP Request: wechat-publisher]                          │
│       ↓                                                     │
│  [Response]                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
        │                    │                    │
        ▼                    ▼                    ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ Claude Code │    │ Claude Code │    │ Claude Code │
│   Skill     │    │   Skill     │    │   Skill     │
└─────────────┘    └─────────────┘    └─────────────┘
```

## ADR-003: 大语言模型选型

### 状态
✅ **已批准**

### 上下文
需要选择主要的大语言模型，驱动各个 Agent。

### 考虑的选项

#### 选项 A: Claude Opus 4.6 (主要)
| 优点 | 缺点 |
|-----|------|
| ✅ 最强推理能力 | 成本较高 |
| ✅ 200K+ 上下文 | - |
| ✅ 输出稳定 | - |

#### 选项 B: DeepSeek-V3 (辅助)
| 优点 | 缺点 |
|-----|------|
| ✅ 成本低 | 推理能力略弱 |
| ✅ 开源友好 | - |

### 决策
**Claude Opus 4.6 为主，DeepSeek-V3 为辅**

### 分配策略
| 任务类型 | 使用模型 | 理由 |
|---------|---------|------|
| 创作写作 | Claude Opus 4.6 | 质量要求高 |
| 审核评分 | Claude Opus 4.6 | 需要深度推理 |
| 简单处理 | DeepSeek-V3 | 成本优化 |
| 数据分析 | DeepSeek-V3 | 结构化任务 |

## ADR-004: 数据存储选型

### 状态
✅ **已批准**

### 决策
**飞书多维表格 + 飞书云文档**

### 理由
1. 结构化数据（选题、草稿、审核记录）→ 多维表格
2. 非结构化数据（成品内容）→ 云文档
3. 易于协作和备份
4. 用户已确认使用飞书

### 表结构
| 表名 | 用途 |
|-----|------|
| content_sources | 采集内容池 |
| topics | 选题池 |
| drafts | 草稿 |
| reviews | 审核记录 |
| publishes | 发布记录 |
| analytics | 数据反馈 |

## ADR-005: 交互方式选型

### 状态
✅ **已批准**

### 决策
**飞书小程序 / 多维表格 + 飞书机器人**

### 理由
1. 用户已有飞书环境
2. 飞书机器人可推送通知
3. 多维表格可做状态管理
4. 支持人工确认按钮

### 交互流程
```
1. 系统推送选题 → 飞书机器人消息
2. 用户点击选择 → 飞书卡片交互
3. 系统推送初稿 → 飞书云文档链接
4. 用户确认 → 飞书机器人按钮
```

## ADR-006: 浏览器自动化选型

### 状态
✅ **已批准**

### 决策
**Playwright (Python)**

### 理由
1. 支持 CDP，反爬虫能力强
2. Python 生态丰富
3. 比 Puppeteer 更现代
4. 比 Selenium 更稳定

## ADR-007: 语音技术选型

### 状态
✅ **已批准**

### 决策
| 功能 | 技术 |
|-----|------|
| 语音识别 (ASR) | Whisper large-v3 |
| 语音克隆 (TTS) | GPT-SoVITS V4 |

### 理由
1. Whisper 开源最强 ASR
2. GPT-SoVITS 中文效果最佳
3. 两者都有 Python 支持

## ADR-008: 图像生成选型

### 状态
✅ **已批准**

### 决策
**DALL-E 3 (主要) + Midjourney (辅助)**

### 理由
1. DALL-E 3 API 稳定，质量一致
2. Midjourney 用于特殊风格需求
3. 信息图使用 Mermaid.js (代码生成)
