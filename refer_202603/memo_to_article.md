## `meme-to-article`

![03252](C:\Users\irisf\Documents\Obsidian Vault\10_geekbang\pre_claw\refer_202603\03252.png)

### 核心设计理念

**传统 case-writer 的逻辑**：观点 → 找案例 → 案例撑观点 **meme-to-article 的逻辑**：主题 → 搜梗 → 梗本身就是观察角度 → 用案例解释梗为什么火 → 梗串联全文

也就是说，文章的骨架不再是"论点-论据"，而是"梗-解读-案例-洞察"。每个大段落的锚点是一个梗，案例是为了解释这个梗背后的深层逻辑。

------

### 七阶段流水线

#### Phase 1: 主题输入 + 全网搜梗

> 来源：meme-search 的搜索引擎

- 用户输入主题（如"AI 替代程序员"）
- 执行 meme-search 核心搜索：
  - 生成 8-12 条多维搜索词（中英混合）
  - 6 策略并行：Web分层时效搜索 / Twitter热梗采集 / 图片搜索 / 平台定向 / Perplexity深搜 / 平台热点趋势
  - 时效三层筛选：24h → 7d → 30d
  - 按热度+时效+相关性排序
- 产出：分类梗库（图片梗 + 文字梗，按趋势/近期/经典分组）

#### Phase 2: 梗驱动的角度设计 + 用户确认

> 新增衔接层（这是合并的关键创新）

AI 分析搜到的梗，提炼 **3 条梗驱动的文章线索**，每条包含：

| 维度               | 内容                                   |
| ------------------ | -------------------------------------- |
| 线索名称           | 如"程序员的自嘲式抵抗"                 |
| 主线梗             | 串联全文的 2-3 个核心梗                |
| 梗的张力           | 这些梗为什么火（背后的情绪/矛盾/荒诞） |
| 需要补充的案例方向 | 为了解释梗需要搜什么案例               |
| 预期文章节奏       | 梗→解读→案例→洞察 的大致章节分布       |

**用户交互 #1**：选线索（或组合/自由发挥）

#### Phase 3: 定向案例搜索

> 来源：case-writer-hybrid 的案例搜索，精简版

- 围绕用户选定的线索，搜索 **支撑梗解读的案例**（不是泛搜，是定向搜）
- 目标：5-8 个候选案例（精于多）
- 6C 评分筛选（对比性/具体性/时效性/可信度/完整性/关联性）
- AI 自动匹配：哪个案例最适合解释哪个梗

#### Phase 4: 三版竞写 + 用户选稿

> 来源：case-writer-hybrid 的竞写引擎，改造为梗驱动结构

三个版本的核心差异不再是"商评/趋势/故事"，而是**三种不同的梗叙事策略**：

| 版本         | 叙事策略                         | 结构               | 风格参考             |
| ------------ | -------------------------------- | ------------------ | -------------------- |
| A 梗解读流   | 梗 → 文化分析 → 商业洞察         | 每节一个梗展开     | GQ/Vice 深度文化报道 |
| B 反转叙事流 | 常识判断 → 梗揭示真相 → 案例验证 | 先立后破，梗做转折 | 半佛仙人 / 回形针    |
| C 连环梗流   | 梗1引出梗2引出梗3                | 梗链条层层递进     | 推特热帖 / 即刻长文  |

每个版本：

- 2000-3000 字
- 内置 Writer-Critic 自审一轮
- 梗融入要求：**每 500-800 字至少出现一个梗**（图片梗/文字梗/热评/段子）
- 六维度自动评分（调整权重，见下表）
- 交叉验证数据一致性

**调整后的六维评分**：

| 维度       | 权重 | 说明                                 |
| ---------- | ---- | ------------------------------------ |
| 梗的融入度 | 25%  | 梗是否自然、密度是否够、是否构成主线 |
| 案例质量   | 15%  | 6C 评分，但权重下降（案例是配角）    |
| 叙事吸引力 | 25%  | Hook、节奏、转折、可读性             |
| 洞察深度   | 15%  | 从梗到洞察的跳跃是否有说服力         |
| 结构清晰度 | 10%  | 段落节奏、视觉锚点                   |
| 金句密度   | 10%  | 自造金句 + 梗金句的密度              |

**用户交互 #2**：对比面板选版本

#### Phase 5: 去AI味 + 润色

> 来源：solo-writer 的去AI味系统

- 套用替换表（开头/连接词/结尾/禁用词全套）
- 注入对话感（你、说白了、讲道理、对吧？）
- 节奏控制（每段不超4句、长短交替、每500字一个视觉锚点）
- 标记图片位置 `![caption](imgN_keyword.png)`
- 自检 checklist
- **用户交互 #3**：审稿，可迭代修改

#### Phase 6: 并行图片生成

> 来源：solo-writer 的图片系统

- 解析所有图片标记位置
- 按 5 种模板选型（对比/流程/概念/边界/数据）
- 并行 background Task 生成信息图
- 统一风格（扁平、2-3色、16:9）

#### Phase 7: 公众号排版 + 封面

> 来源：solo-writer 的排版系统

- 调用 wechat-formatter → 内联CSS + base64图片嵌入
- 生成封面 HTML（主封面 2.35:1 + 朋友圈封面 1:1）
- 最终产出文件：
  - `outputs/article.md`
  - `outputs/wechat/article.html`
  - `outputs/cover.html`
  - `outputs/img*.png`

### 需要你确认的点

**1. 三版竞写的叙事策略**

我设计的三种（梗解读流 / 反转叙事流 / 连环梗流），你觉得这三个分型合理吗？有没有想替换的？

**2. 用户交互 3 次够吗**

- 选线索（Phase 2）
- 选版本（Phase 4）
- 审稿（Phase 5）

**3. 案例的角色**

目前设计是"案例为梗服务"——搜到梗后再定向搜案例来解释梗。这意味着有些文章可能案例很少（如果梗本身就够有说服力的话）。你 OK 吗？还是希望每篇文章至少保证 N 个案例？

完成了。这张流程图完整展示了 "梗到文章" 流水线从输入到产出的 8 个阶段：

**第一行 -- 素材准备层（阶段 1-3）**

- 搜梗引擎：4 路并行搜索全网梗图和文字梗
- 角度设计：AI 设计 3 条差异化写作路线，用户选择方向（人机协同点 1）
- 案例搜索：按路线定向搜真实案例，构建弹药库

**第二行 -- 内容创作层（阶段 4-5）**

- 多版竞写：3 个写手并行竞写 + 六维度自动评分，用户选稿（人机协同点 2）
- 去 AI 打磨：系统性替换 AI 八股句式，输出终稿

**第三行 -- 视觉产出层（阶段 6-8）**

- 信息图生成：并行生成 12 张统一配色信息图
- 微信排版：Markdown 转微信兼容 HTML
- 封面生成：主封面 + 朋友圈封面双版本
